## 1. 確定した構想: 「見られるために置かれた UI」

トップページの方向として、オーナー発案の次の構想が確定した(2026-07-04):

**@otibo/ui のコンポーネントを「操作されるためではなく、見られるために」場に置く。**
道具そのものを鑑賞対象として展示する。工房が自分の道具で自分の家を建てて見せる形の
dogfooding が、そのまま「一人の作り手が道具を丁寧に作っている場所」の証明になる。

この構想が解く問題(対話で確認済み):

- 「何をしているか」の空洞: 主張を文章でなく実在する工学(動くコンポーネント)の展示で示せる。
- 柔と鋭の同居: 場(正典の柔・有機)の上に本物の人工物(精密)が置かれる —
  正典画像の読解「有機的で柔らかいが、人為性も感じる」と構造が一致する。捏造が不要になる。
- 一般訪問者への可読性: UI は誰もが毎日触る日用品。丁寧に置かれていれば誰にでも通じる。
- 未公開プロダクトとの接続: 「これらの道具から、アプリが生まれつつある」という材料と作品の関係。

### 境界条件(硬い制約)

1. **@otibo/ui の宣伝ページにしない**(site purpose intent の non-goal)。インストール手順・
   GitHub バッジ・「design system」という語を出さない。見せるのは「ライブラリ」ではなく
   「この手の仕事の質」。
2. **展示のために新しいコンポーネント族を作り始めたら、それは旧 editorial-ui の死の再演**。
   展示してよいのは実在する core-ui のみ。演出は構成と光(場)だけが担う。
3. 展示の様式の帯域はオーケストレーター見立てで「静物(使われかけの場面の凍結)〜
   生きた静物(操作を受けず自律的に微かに動く)」の間。**オーナー判定は未取得**。

## 2. 制作ラウンドの経緯と死因(要旨)

第 1〜3 ラウンド + 診断プローブ 2 本を実施し、**最終的に全成果物をオーナーが
「AI Slop が前面に出ている」と判定して廃棄**した(成果物は session scratchpad にあり消失予定。
参照価値は低い — 何が死んだかの記録だけ本書に残す)。

- R1(3 案・独立エージェント): 全滅。全案が Noto Serif JP 300 + 均一ベタ地の
  「上品な和文ミニマル」に収束。死因 = 共有した実務翻訳(guide)による均質化 +
  禁止事項だけのプロンプト(臆病はスペクタクルと同格の失敗)+ 制作者が自作を見ていない。
- 診断プローブ: 仮説「人為性(工学・精密)の半身の欠如」= 支持。仮説「hero→products→about
  の骨格自体が借り物」= 支持。両者は相補的。
- R2/R3: 場の呼吸・言葉の事件・脱スキーマ地形はそれぞれ部分的に機能したが、
  総体としては「デザインについてのジェスチャーの集積」(何も測っていない計測ラベル、
  理由を後付けした線)= Slop と判定。**路線(方向)は正しい、はオーナー同意済み**。
- 未解決の仮説: コピーの声(生成文の「静かに解く」系はオーナーの精密な文体と別人。
  本人執筆が最良)/ 物質性(CSS gradient は正典の写実に届かない。第二段階で視覚資産の導入を検討)。
- 訪問者問答の結論: 現状の延長は「信頼される佇まい」までは届くが「他のアプリを触りたい」には
  **製品の実在感**(実画面の断片等)が必要。これはプロダクト成熟待ちの第二段階として予約。

## 3. 次の一手(再検討中)

2026-07-08 追記: 旧「手札カタログ」survey は、調査と選別の練度が不足していたため破棄した。
トップページの目標自体は維持する:

**3 秒で「良い」と感じさせ、30 秒で「一人の作り手が道具を丁寧に作っている場所だ」と信頼させる。
従として、Medo がここで育っていると伝わる。**

次は、目的からアプローチを再設計する。旧カード番号や旧 survey の分類には依存しない。
試作は引き続き小さく始める。まず一つの対象(Button / Toast / Medo に接続する小さな UI 断片など)を
選び、成立を確認してからページ構成へ進む。

**キャプション文はオーナーが執筆する**(Slop 不能な材料。数本の短文)。未執筆箇所は生成せず、
プレースホルダとして明示する。

### 3.1 先に作る判断軸

次の成果物はデザイン案ではなく、**トップページが何を証明すればよいかの判断軸**である。
具体的手法(shader / scroll animation / micro interaction / 画像生成スタイル等)は、この判断軸の後に選ぶ。

- **3 秒で「良い」**: 一瞥で場の質が伝わる。分析を要求せず、名指ししやすい様式に回収されない。
- **30 秒で「一人の作り手」**: 大量生産テンプレートではなく、判断と手入れの痕跡が見える。ただし手書き風・工房風・クラフト美学へ逃げない。
- **「道具を丁寧に作っている場所」**: 完成品広告ではなく、道具を作る態度と精度が見える。UI 部品を飾ること自体は目的ではない。
- **「Medo がここで育っている」**: 未完成を誠実に扱い、プロダクトが生まれつつある気配を伝える。実在しない完成度や未来を宣言しない。

### 3.2 image gen で判定するもの / しないもの

image gen は、色・光・密度・構図・物質感・「一瞥の良さ」の仮説を見るために使う。
一方、web でしか成立しない shader / scroll animation / micro interaction / 状態遷移は静止画で判定しない。
静止画でそれらを表現しようとして「それらしいレイアウト」だけを採用しない。

画像生成の役割は、実装手法の決定ではなく、**場の方向と誤読の早期発見**である。
最終判断はオーナーの視覚判定を信頼する。エージェントは判断を代替せず、比較しやすい仮説と残る未検証点を出す。

### 3.3 体験フロー: 見る → 読む → 確認する

2026-07-09 追記: 前回の失敗は、「見せる」「説明する」「触らせる」が同じ平面に混ざり、
各要素の主語が薄れたことにある。以後のトップページは、体験モードを分離する。

#### First View: 見る

First View は、otibo という名前を、光の場と一緒に記憶させるための入口である。
説明はしない。責務は次の三つに限定する。

- 一瞥で「ここには質がある」と感じさせる。
- otibo という名前を、光の場と結びつけて記憶させる。
- すぐ下の説明へ進むだけの余韻を残す。

禁止:

- タグラインを置かない。
- 説明文を置かない。
- プロダクト名を置かない。
- UI コンポーネントを置かない。
- 操作可能な要素を置かない。
- CTA を置かない。
- 「何をしているか」を First View で解決しようとしない。

#### What otibo is: 読む

このセクションは、otibo が何であるのかを明示的に伝える最初で最後の場所である。
First View が説明しない分、ここで短く意味を渡す。

責務:

- otibo が何の場所かを短く分からせる。
- 読んだ人が、次のプロダクト確認へ自然に進める状態を作る。
- 組織の実体と矛盾しない語り口にする。ただし「一人で作っている」ことを主張の中心にしない。

禁止:

- 理念文・マニフェストを長く書かない。
- 詩的な抽象文にしない。
- 「静かに解く」系の生成文に逃げない。
- 複数人チームのように見せない。
- 成果を大きく見せない。
- プロダクト一覧をここで展開しない。
- `@otibo/ui` をここで説明しない。

#### What otibo makes: 確認する

このセクションは、otibo が現状保有するプロダクトを示す場所である。
製品広告ではなく、otibo が何をしているのかを実例で確認する。

責務:

- otibo が現状保有するプロダクトを示す。
- 各プロダクトが何であるかを短く分からせる。
- 必要な導線(ストアページや詳細へのリンク等)を置く。

禁止:

- 操作デモではない。
- 機能一覧を広告しすぎない。
- 宣伝の毛色を強めすぎない。
- `@otibo/ui` を含めない。これはプロダクトではなく開発ツールに近い。

#### Where otibo is: 所在 / 導線

このセクションは、訪問者が otibo の実在する導線へ進む場所である。
Contact / Legal / 外部ページ等を静かに置く。

責務:

- otibo への連絡手段を示す。
- 法務ページなど、公開上必要な実在の導線を置く。
- プロダクトの外部ページ、ストア、必要な詳細へ進める。

禁止:

- CTA を強めない。
- フッターを二度目の見せ場にしない。
- 連絡フォームなど、まだ不要な機能を増やさない。
- SNS や外部リンクを数合わせで置かない。
- First View の光を大きく再演しない。使う場合も、入口の記憶が薄く戻ってくる程度に留める。

## 4. 制作運用の規律(全滅から確立。エージェント運用者向け)

- 視覚成果物は制作者自身が Playwright スクリーンショット → 自分の目で見る → 正典画像と
  照合 → 直す(最低 2 周)を提出条件とする。レビュアーも必ずレンダリングを見る。
- プロンプトは加速(場を作れ・committed な色)と抑制(事件は一つ)の両輪。禁止リストだけ渡さない。
- 均質化対策: 前ラウンドの収束点(共通既定値)を名指しで禁止。配布文書を案ごとに差別化。
- 調査系の委譲には「サブエージェント起動禁止・自分で検索を実行」を明記する
  (2026-07-04 に多段委譲の連鎖で 40 体超が空転した事故あり)。
- guide(実務翻訳)を制作者全員に渡すと出力が均質化する。判定の権威は常に画像と正典文書。

## 5. プロトタイピング指示の雛形(ワンページ制作を委譲するときの部品 10 個)

良い指示は仕様書ではなくクリエイティブ・ディレクション。以下を順に含める:

1. **対象の再定義(1 文)**: 「これは工房の展示室である。@otibo/ui のコンポーネントは
   操作されるためではなく、見られるために置かれる」。
2. **効果の定義**: 「3 秒で『良い』、30 秒で『一人の作り手が道具を丁寧に作っている場所だ』と信頼」。
3. **材料の実在性**(最重要の Slop 対策): core-ui を改変せずそのまま(新コンポーネント族の
   創設禁止)。コピーはオーナー執筆文のみ(未執筆箇所は生成せずプレースホルダ明示)。
   プロダクトは実名 + 正直なステータス。実在しないもののジェスチャーを描かない。
4. **権威の序列**: 判定は正典画像 + `_docs/reference/Site/visual-canon/reference.md` のみ。
   guide / 再設計中の技法メモは道具。満たすことの目標化は違反。
5. **加速と抑制の対**: 「場を作れ(ベタ地は場ではない)・名指せない暖色に commit・非対称と
   重なりを恐れるな」+「事件は一画面に一つ・暗地/発光/スペクタクル/テクスチャ禁止・
   臆病はスペクタクルと同格の失敗」。
6. **体験順序**: セクション構成(hero/products/about)は渡さない(骨格の借用が死因だった)。
   「0 秒で何が知覚され、0.5 秒で視線がどこへ落ち、30 秒で何を理解して帰るか」の時間軸を渡す。
7. **表現語彙の限定**: 再設計後のアプローチから使う語彙を絞る。語彙を絞ると必然性が生まれる。
8. **細部の密度**: 禁則・孤立行ゼロ / footer の最小文字まで設計対象 / 「理由を言えない要素は削除」。
9. **プロセスの指定**: スクリーンショット → 自分の目で見る → 正典画像と照合 → 直す、最低 2 周。
   複数ビューポート確認。調査を含む場合はサブエージェント起動禁止を明記。
10. **検収の予告**: 「様式名を言えたら失敗」「3 秒で良いか」「分析を要求していないか」で
    判定されると先に伝える。

技術付録: consumer 設定は otibo-ui README 準拠(panda.buildinfo include + importMap、
RSC では flat export)。

## 6. その他のセッション限り情報

- 公開中トップページ(最小実装)のコピーのうち「個人が作るプロダクトブランドです」は
  オーナーが「変なだけ」と判定済み(未修正のまま公開中)。展示構想の実装時に、
  オーナー執筆のコピーで置き換えること。タグライン「日常の小さな『やりにくい』を、
  静かに解く。」も生成文であり、同時に見直し対象になり得る。
- Medo の paywall 調査・backcast README 修正は別セッションのタスクとして進行(otibo-dev 外)。
